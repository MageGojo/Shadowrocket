# 技术设计

## 配置文件策略

项目保留两份配置：

- `config/sr_ad_only.conf`: 上游原始文件，只作为基线和回滚来源。
- `config/custom.conf`: 后续编辑文件，用于导入 Shadowrocket。

后续新增规则优先写入 `custom.conf`，并尽量在文件中使用注释分区，便于查找和回滚。

## Shadowrocket 规则格式

常见规则格式：

```text
DOMAIN-SUFFIX,example.com,Reject
DOMAIN-KEYWORD,example,Reject
IP-CIDR,1.2.3.0/24,Reject
FINAL,DIRECT
```

第三列可以是内置策略 `DIRECT`、`Reject`，也可以是用户配置里的代理组名称。

## 数据库结构

本机 `sr_ad_only.db` 是 SQLite 数据库，表结构包含：

- `config`: FTS3 虚拟表，保存配置项。
- `config_content`: FTS 内容表。
- `config_segdir` / `config_segments`: FTS 索引表。

当前抽样结果显示规则映射如下：

```text
section = rule
name    = DOMAIN-SUFFIX / IP-CIDR / COMMENT
value   = 域名或 IP 段
option  = REJECT / 0 / 2
```

当前记录统计：

- `rule`: 56359 条。
- `general`: 1 条。
- `DOMAIN-SUFFIX, REJECT`: 56205 条。
- `IP-CIDR, REJECT`: 128 条。

## 策略组地区设计

每个平台使用各自“专属地区”，而不是所有 AI 组共用同一份大地区列表。原因与取舍：

- 旧问题：所有 AI 组用同一份 10+ 地区正则，且 `US` 无词界会误配 `Plus`、`Russia` 等带 "us" 的节点；筛选混乱时易默认落到订阅里的“剩余流量/到期”等信息节点。
- 新方案：按各服务真实可用性，给每个组指定专属地区，并收紧正则。
  - OpenAI / Claude / GoogleAI → 🇺🇸 美国（US）
  - Gemini → 🇯🇵 日本（JP）
  - GoogleSearch → 🇭🇰🇹🇼🇯🇵🇸🇬 香港/台湾/日本/新加坡（就近择优）
  - Apple → 🇭🇰🇺🇸 + `DIRECT`（默认直连）
- 正则匹配三选一：旗帜 emoji（如 `🇺🇸`）、中文名（`美国/美國`）、带词界的英文代码（`\bUS\b`）。
  - `\b` 词界避免 `US` 误配 `Plus`；信息节点（剩余流量/官网/到期）因无地区标识被自动排除。
- 局限：若订阅缺少对应地区节点，对应组会为空。此时需按用户实际节点地区调整正则。

### 手动 + 自动 双组结构

每个平台拆成两个策略组，兼顾“手动指定节点”和“自动选最快”：

- `<平台>-Auto`（`url-test`）：在专属地区正则范围内自动测速，选延迟最低的节点。
  - 参数：`url=http://www.gstatic.com/generate_204`、`interval=300`、`tolerance=50`。
  - 因为是测速选活节点，天然不会停在“剩余流量/到期”等死节点上。
- `<平台>`（`select`，首页可见）：第一项为 `<平台>-Auto`（默认自动），其余为该地区的具体节点（手动可选）。
  - 主组名保持不变（OpenAI/Gemini/...），`[Rule]` 引用无需改动。
  - Apple 主组首项保留 `DIRECT`（默认直连），再给 `Apple-Auto` 与 HK/US 节点。

## TUN 配置设计

需求是“给小火箭加 TUN 模型”。结论与取舍：

- TUN 不是节点策略组，也不是规则里的 `PROXY/DIRECT/REJECT`，而是 `[General]` 下的路由参数。
- TUN 的“总开关”由 Shadowrocket App / 系统 VPN 状态控制，配置文件无法用一行“打开 TUN”。
- 配置文件里能写的是 TUN 旁路与系统兼容路由，写入 `custom.conf` 文件开头的 `[General]` 段：
  - `bypass-system = true`：系统流量旁路。
  - `skip-proxy`：内网网段与本地域名直连。
  - `tun-excluded-routes`：TUN 排除的路由（内网、保留地址、组播等），避免内网/本地流量被错误纳入 TUN。
- 采用“改 `custom.conf` 重新导入”方案，而非直接写沙盒 `.db`，原因是直接改数据库风险高、易写坏，且不便回滚。

## 后续修改原则

- 不直接改 Shadowrocket 沙盒内的原始数据库。
- 先改 `config/custom.conf`，确认后再由用户导入小火箭。
- 新规则放在文件靠前位置，以便优先匹配。
- 对可能误杀的广告规则，优先使用注释说明来源和原因。
- 段落顺序保持 `[General]` → `[Proxy Group]` → `[Rule]`，新增段落注意不要与已有段重复。
